## BLOCKERS — fix before deploy

### B1 — TOTP login is decorative; any 6-digit code logs anyone in as anyone
- File: `devdash.html:1494-1502` (`tryLogin()`), `devdash.html:1485-1493` (`pickUser`, `qrUrl`), `devdash.html:1269` (UI label "Demo mode — any 6-digit code works").
- What: `tryLogin()` only checks `/^\d{6}$/.test(this.codeInput)`. No TOTP verification, no server round-trip. The "secret" used for the QR is just `"DEVDASH" + emailPrefix` — deterministic and known to anyone who reads the source.
- Why it matters: Data on this page includes Compass scores, bonuses, dispute reasons, commit-level dev performance. A user picker plus "any 6 digits" is zero auth. Visiting the page as Fahad or Imran requires one click and six random digits. Also violates the Australian Privacy Act 1988 requirement to take reasonable steps to protect personal information.
- Fix: This is a static prototype. For deploy, either (a) put the HTML behind the real Worker at `/` — Worker already enforces real TOTP via `otplib` against AES-GCM-encrypted KV records and only serves `dashboard:latest:<role>` after a valid session — and delete the in-HTML user picker entirely, OR (b) strip this entire login block from the deployed artifact and rely on the Worker's `LOGIN_FORM`. Do **not** ship `devdash.html` as-is to a public URL.

### B2 — KNOWN BUG CONFIRMED: adding contributors to Phonebot 2.0 still shows only Faizan in PM view
- File: `devdash.html:1306` (seed project data), `devdash.html:1340-1349` (`get devs`), `devdash.html:1552-1556` (`sortedDevsForPm`), `devdash.html:995` (contributor_emails input).
- What: Phonebot 2.0's seed row has `contributor_emails: []` and only `owner_email: faizan@...`. `get devs` is reactive (recomputes on every access) — the bug is **not** reactivity. The bug is **persistence**: when the user edits `p.contributor_emails` in the Settings tab, the `@change` handler updates `p.contributor_emails` in memory (line 995) but **does not call `saveConfig()`**. Every other settings input requires clicking the explicit "Save changes" button (line 998) which *does* persist via `localStorage.setItem('devdash_config', …)`. If the user just types new emails and navigates away (or PM switches tabs), the edit is lost.
- Additionally: the "Save changes" button is only inside the Projects-tab per-project card. But reactivity **is** correct — the in-memory edit *should* immediately propagate to the PM view within the same tab switch, because `devs` is a getter and `sortedDevsForPm` filters off it on every render. So the user-reported "added contributor, still shows only Faizan" means **either** (a) they never clicked Save and the field reverted, **or** (b) a prior `localStorage.devdash_config` (with old `contributor_emails: []`) is overwriting the in-memory default at `init()` line 1474-1477.
- Why it matters: This is the specific bug Fahad flagged, and it has two root causes.
- Fix (both needed):
  1. On line 995, add `saveConfig()` to the `@change` handler: `p.contributor_emails = $event.target.value.split(',').map(s => s.trim()).filter(Boolean); saveConfig()`.
  2. Add a **schema migration** or **cache-bust** on `init()`: if the stored config's version doesn't match the hard-coded default, discard it. Currently `init()` blindly overwrites `this.config` from localStorage with no version check — so any prior test session with old data silently overrides the in-source default forever.
  3. Consider auto-save on all settings @change handlers (user-pattern: they type and expect it saved).

### B3 — Settings tab is visible to CEO + PM without any server-side check
- File: `devdash.html:1505` (`VISIBILITY` object).
- What: Visibility matrix is client-side only. Any dev who opens devtools can run `Alpine.$data(document.body).activeRoleTab = 'settings'` and see/modify every user, every reward amount, every compass threshold — including their own bonus.
- Why it matters: Users explicitly includes Compass scores, AUD bonus amounts, team pool config. Dev can self-promote their bonus by editing `config.rewards.per_direction_aud` or their own `compass` mock data. Fahad said "leaks between roles are a critical bug."
- Fix: Gate `activeRoleTab = 'settings'` and `config.rewards.*`, `config.users`, audit log on a server-side role check. For the current static HTML this is impossible — the fix is either move writes behind the Worker (with role verification from the session KV) OR accept this is a prototype and never deploy it publicly. Add a `<noscript>`-style warning at the top of the file: `<!-- PROTOTYPE ONLY — DO NOT DEPLOY -->`.

### B4 — Dev role can see every other dev's Compass numbers + commits + bonus
- File: `devdash.html:834-860` (dev-picker fallback), `devdash.html:1567-1573` (`devViewTarget`), `devdash.html:1505` (VISIBILITY dev: `['dev','qa','qa_auditor']`).
- What: When a dev logs in, `devViewSelected` is set to their own email (line 1500). But there is **no guard preventing a dev from changing `devViewSelected`**. The dev-picker block at 834 is only gated by role `!== 'dev'`, but nothing stops a dev from doing `Alpine.$data(document.body).devViewSelected = 'usama@phonebot.com.au'` in the console and seeing Usama's full compass + reward amount + commits. The Python-side `role_views.py:46-48` filter *does* strip other devs' data for the dev role — but this HTML ignores that entirely and renders `devMockData` client-side for **all** devs.
- Why it matters: Cross-dev pay/performance visibility. Compass breakdowns are tied to bonuses (`rewardHeadline` at 1602 computes `$N * per_direction_aud`). Peer-compass visibility is supposed to be controlled by `config.scoring.visibility.peer` (line 1069) but that setting isn't read anywhere in the render logic — grep for `scoring.visibility.peer` returns zero hits outside the settings form.
- Fix: Respect `config.scoring.visibility.peer`. When a dev is logged in, `devs` getter should return only `currentUser.email` (or shapes-only / nothing, per setting). Mirror the server-side `role_views.py` filter behavior. Also hide the dev-picker template entirely when `currentUser.role === 'dev'`.

### B5 — `pmSummaryHtml()` uses `x-html` with interpolated dev display names — stored XSS
- File: `devdash.html:427` (`x-html="pmSummaryHtml()"`), `devdash.html:1589-1596` (pmSummaryHtml implementation).
- What: The function returns a literal template string embedding `top.displayName` verbatim into HTML, which is rendered via `x-html` (= `innerHTML`). `displayName` is user-editable in the Settings → Users tab without any sanitization. A PM or CEO can set a dev's displayName to `<img src=x onerror=alert(1)>` and it executes on every PM view render.
- Why it matters: In a 6-person tool this is low exploitation risk, but combined with B3 (settings open to any dev via devtools) a dev could self-XSS their own displayName to run scripts in the PM/CEO session cookie context. Also violates basic defense-in-depth.
- Fix: Either (a) change to `x-text` and drop the `<strong>` styling, or (b) build the HTML via DOM (`document.createElement`) and set `textContent`, or (c) escape interpolated values with a tiny helper: `const esc = s => String(s).replace(/[&<>"']/g, c => ({'&':'&amp;','<':'&lt;','>':'&gt;','"':'&quot;',"'":'&#39;'}[c]))`.

### B6 — `wrangler.toml` has unfilled KV id `<FILL_AFTER_KV_CREATE>` — Worker will not boot
- File: `worker/wrangler.toml:9`.
- What: `id = "<FILL_AFTER_KV_CREATE>"`. A deploy with this placeholder will fail at `wrangler deploy`, but if it doesn't fail (CI bypasses validation), every `env.DASHBOARD_KV.get()` call will throw "no such binding."
- Why it matters: Silent auth failure pattern. Worker's `handleRequest` calls `getSession` → `kv.get` → exception → falls through to `loginHtml()` with no error surfaced to the user. They'll think their TOTP is wrong when KV is actually unreachable.
- Fix: Before deploy, run `wrangler kv:namespace create DASHBOARD_KV` and paste the returned id. Add a CI check that grep-fails if `<FILL_AFTER_` is still in the file. (Note: codebase is migrating to AWS per user prompt — but if this Worker is shipped as a stop-gap, fix this first.)

## HIGH — phase-1 fixes

### H1 — `topPerformer()` throws on empty `devs` list
- File: `devdash.html:1586`.
- What: `this.devs.reduce((a,b) => …)` without initial value crashes on empty array with `TypeError: Reduce of empty array with no initial value`. If a CEO deactivates all devs or a fresh project has zero contributors, CEO view breaks and `pmSummaryHtml()` (which calls `topPerformer()`) throws.
- Fix: `return this.devs.length ? this.devs.reduce(...) : { displayName: '—', summary: '', compass: {velocity:0,craft:0,reliability:0,drive:0} };`

### H2 — `worstCaseLaunch()` parses ad-hoc `"Aug 14, 2026"` strings and will emit NaN
- File: `devdash.html:1578-1584`.
- What: `new Date("Aug 14, 2026")` works in V8 but is non-standard per ECMA and returns Invalid Date in some engines/locales. For project `leg` the filter excludes `forecast_launch === 'Ongoing'`, good — but `new Date("—")` (the empty project default at line 1650) returns Invalid Date; if user adds a project and forgets a forecast, `daysDelta` becomes NaN and the banner shows "NaN d late".
- Fix: Store `forecast_launch` as ISO `"2026-08-14"`. Display-format at render time. Also guard: `if (isNaN(daysDelta)) daysDelta = 0;`.

### H3 — `saveConfig()` records the audit log AFTER writing — audit trail lies about who changed what
- File: `devdash.html:1480-1483`.
- What: `this.auditLog.unshift(...)` pushes to an in-memory array, but `auditLog` is never persisted to localStorage. On any reload the audit log reverts to the 3 hard-coded entries on lines 1450-1454. Fahad's entire change history vanishes every session.
- Fix: Add `this.save('auditLog')` after `unshift`, and load it in `init()`'s foreach loop (line 1474 — currently covers `bugs, auditFindings, featureRequests, config, clockEntries` but not `auditLog`, `disputes`, or `pmAssessments`). Also add `disputes` and `pmAssessments` to the load loop — they save to localStorage (lines 1691, 1684) but never load back.

### H4 — `submitOffProject()` mutates `devMockData` but never calls `save()`
- File: `devdash.html:1674-1681`.
- What: User logs off-project hours, the data updates in memory, form closes. On next reload, hours revert to the seed. Also, `devMockData` is never loaded from localStorage either.
- Fix: Add `this.save('devMockData')` (and add `devMockData` to the init load list). Better long-term: all stateful data should pass through a single `save(key)` helper, and `init` should reload every stateful key.

### H5 — `submitFeature()` crashes if `currentUser` is undefined
- File: `devdash.html:1669`.
- What: `requester: this.currentUser.displayName` — but the feature-request button is shown only for ceo/pm (line 1199 gate). Still, if a dev sneaks through via devtools-forced `showFeatureForm = true`, or session expires mid-form, this throws.
- Fix: `requester: this.currentUser?.displayName || 'unknown'`.

### H6 — Handoff parser silently drops entries whose header uses ASCII `-` on both sides
- File: `scripts/dashboard/handoff_parser.py:28`.
- What: Regex `r"^##\s+(\d{4}-\d{2}-\d{2})\s+\d{2}:\d{2}\s+[—-]\s+([^/\n]+?)(?:\s*/.*)?$"`. This matches `— author` (em dash) OR `- author` (hyphen). But the author capture `([^/\n]+?)` is non-greedy + forbids `/` — if a human writes `## 2026-04-25 18:00 — Faizan` (no slash context), `([^/\n]+?)` matches the minimum "F" then the optional non-capturing group tries to match rest. This works in practice because of the `$` anchor and non-greedy, but is fragile. More importantly: **any line where the author contains a slash like "Fahad/Claude" ends up with author == "Fahad"**, dropping the context.
- Why it matters: Handoff entries drive the merit reliability score. Silent truncation of "Fahad/Claude" → "Fahad" means PM handoff attribution is wrong.
- Fix: Simpler pattern: split on first `—` or `-` after the time, take everything before the first ` / ` or newline. Or just keep everything after the separator as the author+context string.

### H7 — Handoff parser `section` never resets on empty lines between sections
- File: `scripts/dashboard/handoff_parser.py:60-81`.
- What: `flush()` only fires when a new label (`CLOSED:`, etc.) is encountered, or on header match. If a handoff entry is missing `OPEN:` entirely, content intended for the *next* entry's header is appended to the previous entry's last section buffer — until the header regex finally matches. That mostly works because headers are on their own line, but any line like `## Something else` in the middle of content will break parsing.
- Fix: On blank line → flush current section. Also unit-test with a handoff missing one of the four sections.

### H8 — `git_reader.read_commits_since` file-list parsing is quadratic and confused
- File: `scripts/dashboard/git_reader.py:38-82`.
- What: The code splits on `<<<END>>>` then for each chunk *also looks into the next chunk* to append files. This double-consumes: commit N's files appear on commit N (from `lines[1:]`) AND commit N's files get re-read from `chunks[i+1]` before the next commit's header. `git log --pretty=format:FMT --name-only` outputs `FMT_LINE\nfile1\nfile2\n\nFMT_LINE_NEXT\n...` — the files are **already** in the same chunk as their commit's FMT_LINE once you split on END. The secondary append is duplicative and will include the NEXT commit's files on the PREVIOUS commit until the `if _FIELD in line: break` kicks in — but only checks for FIELD delimiter, not END delimiter, which never appears on a filename line anyway. Result: unclear what actually gets populated.
- Fix: Use `--format=%H%x00%an%x00%ae%x00%aI%x00%B%x1e` with `%x1e` record separator, then a proper state machine. Or use `pygit2` / `GitPython`. Current tests only exercise the 1-commit case so they pass but don't exercise this logic.

### H9 — `git_reader` does not trap `subprocess.CalledProcessError` for corrupted repos
- File: `scripts/dashboard/git_reader.py:30-33`.
- What: `subprocess.run(..., check=True)` raises on non-zero exit. Deleted `.git`, dangling worktree, or a repo that git rejects as "not a valid git repository" all raise `CalledProcessError`, not `FileNotFoundError`. Caller can't distinguish "repo gone" from "arg parse error."
- Fix: Wrap in try/except, classify: rebirth-as-FileNotFoundError for missing, GitReadError for transient.

### H10 — CSV-encoded list fields in Settings silently drop empty strings and mis-parse trailing commas
- File: `devdash.html:994` (repos), `devdash.html:1103` (unlock_thresholds), `devdash.html:1147` (severity_labels), `devdash.html:1148` (urgency_labels), `devdash.html:1158` (digest_recipients).
- What: `$event.target.value.split(',').map(s => s.trim())`. Typing `HIGH, FUNCTIONAL, VISUAL,` (trailing comma) yields `['HIGH','FUNCTIONAL','VISUAL','']`. That empty string then shows up as a dropdown option in the bug-severity select, and `filteredBugs` comparisons break.
- Why it matters: A silent data-corruption bug — labels drift as users tweak them. `unlock_thresholds` (line 1103) calls `parseInt` which on empty string returns NaN — worse.
- Fix: Always chain `.filter(Boolean)` (the contributor_emails field on 995 does this — make it uniform).

### H11 — `$event.target.value` doesn't update when the user types then blurs via Tab into another field
- File: `devdash.html:994, 995, 1103, 1147, 1148, 1158`.
- What: These use `@change` not `@input`. `@change` fires on blur. If the user types in the repos field then clicks "Save changes" without blurring first (mouse click on the button), some browsers fire click before change → save reads stale value. Safari historically has the worst behavior here.
- Fix: Use `@input` for text inputs, or swap to `x-model` with a computed wrapper that serializes the array. Also helps for the contributor_emails field Fahad is complaining about.

### H12 — `suggestDevForBug` returns `'Unassigned'` string, `submitBug` assigns it as `assigned_to`
- File: `devdash.html:1597-1601`, `devdash.html:1657-1661`.
- What: Bug's `assigned_to` field ends up as the string `"Unassigned"`. `bugsForDev('Unassigned')` returns nothing (nobody is named Unassigned) — so an unassigned HIGH bug becomes invisible in every dev view, counts in CEO stats (line 402) but nobody is alerted.
- Fix: Use `null` for unassigned, render `—` in the UI, and surface "unassigned HIGH bugs" separately on CEO view.

### H13 — PM's "disagreement flagged" trigger is based on assessment *length*, not content
- File: `devdash.html:1683`.
- What: `disagreement_flagged: this.newPmAssessment.length > 50`. Every 51+ character assessment gets a disagreement flag regardless of what it says. That flag eventually drives the alert email in the Worker (`formatAlertEmail` checks `disagreements.length > 0`). Every moderately-written PM note will therefore spam Fahad.
- Fix: The disagreement flag should come from the upstream Python pipeline comparing PM's text against the AI-generated assessment — not from string length. For now: flip to `false` until the comparator exists.

### H14 — Worker login returns the same 200 `loginHtml` for "unknown user", "bad code", and "user record corrupted" — reveals enumeration only via timing
- File: `worker/src/routes.ts:96-126`.
- What: The code does carefully use `"Login failed."` for both unknown user and bad code (good, anti-enumeration). But the `decryptSecret` call runs only when the user exists — so the response time for an existing-email + bad-code is meaningfully longer than for a nonexistent email. Attacker can enumerate valid staff emails via timing.
- Fix: Always run a dummy decrypt against a constant ciphertext when user lookup fails, so timing is flat. (Low exploitation value for a 10-person staff list with known `@phonebot.com.au` pattern, but free to fix.)

### M9 — `visible_tabs` list in Python `role_views.py` excludes `'settings'`, but `devdash.html` includes it for CEO/PM

### M10 — No tests exist for the Cloudflare Worker TS code
- Files: `worker/src/*.ts`.
- What: No `worker/test/` folder, no `vitest`/`miniflare` config. `decryptSecret` parity with Python `encrypt_secret` is crucial — if the formats drift, every login breaks.

### M11 — `worker_push.py` `_kv_put` writes with no retry and no content-type header
- File: `scripts/dashboard/worker_push.py:19-25`.
- What: No retry on 429/503. Cloudflare KV is eventually consistent; a rate-limit is plausible on a batch push of 5+ keys.
- Fix: Wrap in retry-with-backoff (3 attempts, 1-2-4s). Also a 30s timeout would be wise.

### M12 — `encrypt_secret` / `decrypt_secret` use base64url for encrypted, standard base64 for key — easy to mix up
- File: `scripts/dashboard/totp_provision.py:53`, `worker/src/totp.ts:18-27`.
- What: Python emits base64**url** nonce+ct, key is passed as a separate 32-byte raw arg. Worker side: `keyB64` is standard base64 (not url-safe), encrypted blob is base64url. The Worker docstring notes this. It's correct but fragile — a sysadmin who pipes `openssl rand -base64 32 | sed 's|/|_|g'` will set a url-safe key and decryption silently returns gibberish (AES-GCM raises on tag mismatch, good).
- Fix: Document in a tiny ops check: `node -e 'console.log(atob(process.env.TOTP_ENCRYPTION_KEY).length)'` must print 32.

### M13 — Dashboard.config.yaml `uploads_repo_path` is an absolute local path in a file that's likely checked in
- File: `dashboard.config.yaml:2`.
- What: `/Users/adminadmin/Downloads/phonebot revamp/dev dashboard/dev-dashboard-inputs` is specific to this machine. If the file is committed, CI and teammates will fail to read uploads.
- Fix: Either put it in `.gitignore` (it isn't), or use a relative path / env var `${UPLOADS_PATH:-./dev-dashboard-inputs}` resolved in code.
